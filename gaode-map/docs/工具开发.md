# 工具开发设计

## 1. 设计目标

本项目中的工具不是给人直接点按钮用的功能块，而是给 Agent 调用的标准化能力单元。

工具设计的目标是：

- 把现有系统能力稳定地暴露给 Agent
- 让 Agent 可以按任务组合调用，而不是只能整块调用大模块
- 降低工具之间的耦合，提升可复用性、可测试性和可缓存性
- 让执行链路、失败点、证据来源都可追踪

工具不应直接等同于某个业务模块，也不应细碎到只剩普通程序辅助函数。

合适的工具应该满足：

- 有明确输入
- 有稳定输出
- 有单一职责
- 能被 Agent 当成一个领域动作来调用
- 能声明前置条件和产出物

---

## 2. 工具分层设计

建议把工具分成 5 层，而不是简单按模块目录拆分。

### 2.1 L1 原子工具层

L1 是最小可执行的领域能力单元，直接服务 Agent 编排。

特点：

- 单一职责
- 输入输出稳定
- 可独立测试
- 不依赖复杂上下文推理

例如：

- 获取当前分析范围
- 拉取指定范围 POI
- 生成 H3 网格
- 计算路网汇总指标
- 提取关键证据

### 2.2 L2 组合工具层

L2 是由多个原子工具组成的短流程能力，用于覆盖高频任务。

特点：

- 面向一个明确分析任务
- 由少量 L1 工具串联组成
- 仍然保持较强可解释性

例如：

- 供给扫描
- H3 密度分析
- 路网可达性扫描

### 2.3 L3 Harness 运行层

L3 是轻量 runtime / harness 层，不做业务计算，但负责支撑 Agent 的执行循环。

职责：

- 管理 tool registry
- 校验前置条件
- 控制权限模式
- 记录 execution trace
- 控制最大步数和中断条件
- 在工具执行前后运行 hook

它不决定业务目标，但决定执行是否安全、是否合规、是否可追踪。

### 2.4 L4 Agent 编排层

L4 不是工具层本身，而是 Agent 的 `runtime / llm tool loop / executor / auditor`。

职责：

- 判断该调用哪些工具
- 决定调用顺序
- 判断缺不缺输入
- 判断要不要补跑下一步

它不实现底层业务，只编排工具。

### 2.5 L5 输出层

L5 用于把执行结果加工成 Agent 可输出的结构化结果。

例如：

- 证据提取
- 风险提示生成
- 结果归一化
- 卡片生成

这一层本质上也是工具，但更偏“结果处理”。

## 3. 推荐分类方式

如果从 Agent 使用视角来看，工具最适合按“能力类型”分类，而不是按代码目录分类。

| 一级分类 | 二级分类 | 作用 | 是否建议原子化 | 典型输入 | 典型输出 |
| --- | --- | --- | --- | --- | --- |
| 信息获取 | 上下文读取 | 读取当前 analysis 状态、历史快照、已有结果 | 是 | 会话 id、analysis snapshot | 当前 scope、已有摘要、历史结果 |
| 信息获取 | 数据获取 | 拉取外部或内部数据 | 是 | polygon、type、source | POI 列表、人口摘要、夜光摘要 |
| 执行动作 | GIS 计算 | 触发空间分析和地图计算 | 是 | polygon、pois、mode、time_min | isochrone、h3、road 结果 |
| 执行动作 | 业务计算 | 调用现有业务模块做统计、聚合、评分 | 是 | 原始数据、筛选条件 | 分类汇总、供给统计、候选区域 |
| 结果处理 | 证据提取 | 从执行结果中提取能支撑结论的事实 | 是 | h3/road/poi 结果 | evidence 列表 |
| 结果处理 | 结果归一化 | 把不同模块结果转成统一结构 | 是 | 任意模块输出 | 标准 result payload |
| 结果处理 | 输出构建 | 生成卡片、摘要、建议 | 可适度组合 | result、evidence、warnings | summary/risk/recommendation 卡片 |
| 控制辅助 | 校验与约束 | 检查输入、前置条件、执行风险 | 是 | tool input、上下文 | 校验结果、风险标记 |

这个分类方式的核心优点是：

- Agent 容易理解
- 适合做工具注册表
- 适合做前置条件检查
- 适合做缓存和补跑

## 4. 推荐层级结构

建议最终采用下面这套层级：

| 层级 | 名称 | 作用 | 设计原则 | 示例 |
| --- | --- | --- | --- | --- |
| L1 | 原子工具层 | 暴露最小领域能力 | 单一职责、稳定输入输出 | `fetch_pois_in_scope` |
| L2 | 组合工具层 | 封装高频小流程 | 少量 L1 串联，可复用 | `run_poi_supply_scan` |
| L3 | Harness 运行层 | 管理执行控制和治理 | 不做业务实现，只做 runtime 控制 | `runtime` / `registry` |
| L4 | Agent 编排层 | 选择并调度工具 | 不做业务实现，只做决策 | `runtime` / `executor` / `auditor` |
| L5 | 输出构建层 | 组织结果、证据、风险 | 面向 Agent 输出协议 | `build_summary_card` |

建议规则：

- L1 是主力层，优先建设
- L2 只做高频任务，不要泛滥
- L3 不直接写业务逻辑，只做 runtime 和治理
- L4 不直接写业务逻辑，只做编排
- L5 输出必须统一协议

## 6. 原子工具设计原则

### 6.1 原子工具的标准

一个能力如果满足下面条件，就适合做原子工具：

- 它是一个明确领域动作
- 它可以独立执行
- 它的输入不会依赖大量隐式状态
- 它的输出可以被其他工具继续消费
- 它能独立做单测

例如：

- `fetch_pois_in_scope`
- `summarize_poi_categories`
- `build_h3_grid_from_scope`
- `compute_h3_density_metrics`
- `compute_road_syntax_summary`
- `extract_top_cells`

### 6.2 不适合直接做 Agent 工具的东西

下面这类能力不要直接暴露成 Agent 工具：

- `merge_two_lists`
- `format_chart_data`
- `get_value_from_dict`
- `normalize_string`

这些是普通内部函数，不是领域动作。

Agent 工具应该是“领域原子能力”，不是“程序原子能力”。

## 7. 每个工具至少要定义什么

每个工具建议统一定义以下字段：

| 字段 | 含义 |
| --- | --- |
| `name` | 工具唯一标识 |
| `description` | 工具用途说明 |
| `category` | 所属类别，如信息获取/执行动作/结果处理 |
| `layer` | 所属层级，如 L1/L2/L4 |
| `input_schema` | 输入参数结构 |
| `output_schema` | 输出结果结构 |
| `requires` | 前置条件 |
| `produces` | 执行后产出的对象 |
| `readonly` | 是否只读 |
| `cost_level` | 成本等级 |
| `risk_level` | 风险等级 |
| `timeout_sec` | 默认超时 |
| `cacheable` | 是否可缓存 |
| `hooks` | 可绑定的前后置 hook |
| `run()` | 执行函数 |

建议所有工具统一返回：

| 字段 | 含义 |
| --- | --- |
| `tool_name` | 当前工具名 |
| `status` | `success/failed/skipped` |
| `result` | 主结果 |
| `evidence` | 关键证据 |
| `warnings` | 警告信息 |
| `artifacts` | 可供后续工具继续消费的中间产物 |

## 8. Tool Registry 设计

所有工具必须注册到统一的 `tool registry`，而不能由 Agent 直接扫描代码目录。

Tool Registry 的职责：

- 暴露工具列表
- 暴露工具元数据
- 统一工具 schema
- 统一前置条件检查
- 统一风险等级与权限标签

建议规则：

- LLM tool loop 只能从 registry 中选择工具
- executor 只能执行 registry 中注册过的工具
- 未注册能力不能被 Agent 直接调用

这部分就是本项目借鉴 harness 思路后最核心的一层。

---

## 9. Tool Lifecycle Hook 设计

建议为每个工具调用增加统一生命周期 hook。

### 9.1 `pre_tool_use`

执行前做：

- 输入校验
- 上下文校验
- 前置条件检查
- 权限模式判断
- 风险确认判断
- execution trace 记录

### 9.2 `post_tool_use`

执行后做：

- 标准化输出
- 证据提取
- warnings 整理
- artifacts 回写
- 工作记忆更新
- execution trace 更新

这样工具不只是“调用一次函数”，而是被 runtime 管理的一次完整生命周期。

---

## 10. 工具权限分级

建议给工具增加统一权限分级：

| 权限级别 | 含义 | 示例 |
| --- | --- | --- |
| `safe` | 只读或轻量处理，可自动执行 | 读取已有结果、证据提取 |
| `normal` | 常规计算，可默认自动执行 | 等时圈、POI、H3 常规分析 |
| `expensive` | 昂贵或耗时计算，需要 guarded 模式控制 | 大范围路网句法、外部高成本拉取 |

权限分级由 harness runtime 消费，而不是由模型自己决定。

## 12. 推荐的目录组织

建议在 `modules/agent` 下这样组织：

```text
modules/agent/
├── schemas.py
├── runtime.py                # harness runtime
├── tools.py                  # 工具注册表
├── plan_steps.py             # follow-up/test 复用的标准步骤工厂
├── tool_adapters/
│   ├── scope_tools.py
│   ├── poi_tools.py
│   ├── h3_tools.py
│   ├── road_tools.py
│   ├── population_tools.py
│   ├── nightlight_tools.py
│   └── output_tools.py
├── executor.py
├── auditor.py
└── synthesizer.py
```

设计原则：

- `tool_adapters/*` 负责把现有模块能力工具化
- `tools.py` 只负责注册和暴露
- `runtime.py` 负责 hook、governance、trace、llm tool loop 控制
- `plan_steps.py/executor.py` 不直接碰底层模块

## 13. 最终建议

对本项目来说，最合适的工具层级不是“按模块暴露”，而是：

1. 信息获取工具  
2. 执行动作工具  
3. 结果处理工具  
4. Harness 运行层  
5. Agent 编排层

在这之下，再按 L1 原子工具、L2 组合工具做细分。

最重要的原则是：

- 大模块保留业务实现
- Agent 只看到工具化后的能力
- 工具优先设计为领域原子能力
- 所有工具统一由 registry 管理
- 所有工具调用统一经过 hook 和 governance
- 组合工作流由 runtime / executor / auditor 负责拼接

这样后续无论是做自动执行、补跑、回退、审计，还是做缓存和测试，都会更稳。
